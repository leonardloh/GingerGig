<objective>
Implement batch ASR for `ms-MY` and `ta-IN` using **boto3** Transcribe in `ap-southeast-1`, presigned S3 `PUT` for the audio bucket, `POST /api/v1/voice-to-profile/batch` that returns `{"jobId","status":"pending","estimatedSeconds"}` immediately, background coroutine that uses a **fresh** `AsyncSession` from `get_sessionmaker(app.state.engine)` (never the request's `get_db` session) to run `StartTranscriptionJob`, poll `GetTranscriptionJob`, read transcript, call `extract_listing`, and persist on `VoiceSession` with `mode="batch"`, `audio_s3_key`, `transcribe_job_name` filled. `GET /api/v1/voice-to-profile/batch/{job_id}` returns `ready` with `listing` when done. `POST /api/v1/voice-to-profile/audio-upload-url` for presign. Add `tests/test_voice_batch.py` with boto3/Transcribe/S3 fully mocked; no real AWS in CI.
</objective>

<threat_model>
## Security Threat Model

**Assets:** S3 presigned write URLs, batch transcripts, `voice_sessions` rows, elder data isolation.

**Threats and mitigations:**
- **S3 key injection:** require `s3Key` to match prefix `elders/{elder_id}/` or a pattern validated in router before accepting batch job (concrete pattern in `action`).
- **IDOR on GET status:** `SELECT` `VoiceSession` by `id` and `elder_id == current_user.id` or 404.
- **Request-scoped session misuse:** static review: background function must not capture `db: AsyncSession` from `Depends(get_db)`.
- **Extraction failure:** background work stores the safe error string `Listing extraction failed`; `GET /voice-to-profile/batch/{job_id}` maps that terminal state to `HTTPException(502, detail="Listing extraction failed")` so the client-visible API honors VOICE-07.
</threat_model>

- `POST /voice-to-profile/audio-upload-url` body: `contentType: str` (e.g. `audio/wav`), `language: "ms-MY" | "ta-IN"`. Return `201` or `200` with `uploadUrl`, `s3Key`, `expiresIn`. Elder-only.
- `POST /voice-to-profile/batch` body: `s3Key`, `language: "ms-MY" | "ta-IN"`. Validate `s3Key` contains `f"elders/{str(current_user.id)}/"` (or the prefix built by `build_audio_key_elder`); else `403` or `400` with `detail` ascii.
- For POST body use schema `VoiceBatchRequest`; response `VoiceBatchSubmitResponse` with `jobId: UUID` (use `VoiceSession.id` as the job id), `status: "pending"`, `estimatedSeconds: 10` (int literal constant ok).
- Schedule work with `fastapi.BackgroundTasks` or `asyncio.create_task(app.state, ...)`; the async function must:
  1) `sm = get_sessionmaker(app.state.engine); async with sm() as session:` for **all** DB work.
  2) Re-load `VoiceSession` by id, set `transcribe_job_name`, `audio_s3_key = s3Key`, status `transcribing`, commit.
  3) Build `s3://{bucket}/{key}`; call `start_batch_job` + poll; set `transcript`; call `await extract_listing(...)`; set `listing_draft` JSON, status `ready`, or on failure `failed` and `error` message; commit.
- **Never** pass the request `db` from `get_db` into the background function.
- `GET /voice-to-profile/batch/{job_id}` where `job_id` is UUID: load row, if `elder_id != current_user.id` then `HTTPException(404, detail="Not found")` to avoid IDOR. Response models:
  - pending: `status` in `pending` | `transcribing` | `extracting` (match `VoiceSessionStatus` in DB) with `jobId` string from UUID
  - ready: add `listing` (serialized `ListingDraft`)
  - failed due to persistent Qwen validation/parsing: raise `HTTPException(status_code=502, detail="Listing extraction failed")` so the global handler returns `{"status":502,"message":"Listing extraction failed"}`.
  - failed due to non-Qwen batch/transcribe errors: return `200` with `{ "status": "failed", "jobId": "...", "message": "<safe string>" }` unless 04-05 centralizes additional error mappings.
Return **200** for pending/ready/non-Qwen failed states; use `502` for terminal extraction failure only. Document `ready`, `failed`, and extraction-failed behavior in docstring.
