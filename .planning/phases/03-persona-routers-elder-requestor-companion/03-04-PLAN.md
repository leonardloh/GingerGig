<objective>
Replace the requestor router stub with DB-backed search and booking endpoints. Search must return seeded `matchScore`, `matchReason`, and `distance` from Postgres, with deterministic fallback only when those seeded DB fields are null. This plan must not call live Qwen/DashScope.

The top-level `/api/v1/listings/{listingId}` detail route is owned by `03-03-PLAN.md` because it belongs on the existing no-prefix router; this plan only keeps requestor search/bookings aligned with that contract and must not modify `backend/app/routers/elder.py`.
</objective>

<threat_model>
## Security Threat Model

**Assets:** requestor booking history, active listing catalog, elder public profile snapshots, seeded match explanations.

**Threats and mitigations:**
- **Role bypass:** requestor booking endpoints require `current_user.role == "requestor"` and return `403` for elders/companions.
- **Inactive listing booking:** `POST /requestor/bookings` must return `404` when the listing is missing or inactive.
- **Cross-requestor data exposure:** `GET /requestor/bookings` filters only `requestor_user_id == current_user.id`.
- **AI/service leakage:** no requestor route or shared helper may import/call Qwen, DashScope, OpenAI, Redis/Tair, KYC, or voice services.
- **Seed contract drift:** `matchScore` and `matchReason` come from listing DB columns first; fallback is deterministic and local.
</threat_model>

- `amount = first menu item price` if available, otherwise listing price
- `currency = "MYR"`

Flush and return the denormalized `Booking` response. Do not require frontend payload changes.
