<objective>
Replace the elder router stub with protected DB-backed elder endpoints for listings, listing updates, top-level listing detail, bookings, booking response transitions, and earnings summary. Endpoints must preserve frontend route paths and response shapes while enforcing only the basic role/ownership rules agreed for the quick demo.
</objective>

<threat_model>
## Security Threat Model

**Assets:** elder-owned listings, elder booking queue, denormalized requestor snapshots, earnings totals.

**Threats and mitigations:**
- **Non-elder or wrong elder access:** every elder-prefixed route must require role `elder` and `elderId == current_user.id`.
- **Listing tampering:** `PATCH /listings/{listingId}` must load the listing and require `listing.elder_user_id == current_user.id`; non-owners get `403`.
- **Booking tampering:** `POST /bookings/{bookingId}/respond` must validate ownership through the booking's listing.
- **Invalid booking transition:** only `pending -> confirmed` or `pending -> cancelled` is allowed; non-pending states return `409`.
- **Earnings leakage:** earnings summary must aggregate only bookings for listings owned by the current elder.
- **Listing detail leakage:** top-level listing detail returns active listings to requestors, an elder's own inactive listings to that elder, and no unlinked companion data.
</threat_model>
