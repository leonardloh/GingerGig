    ```markdown
    # ApsaraDB PostgreSQL Provisioning Runbook (Phase 1, D-11/D-23)

    > **Status:** Manual one-time setup. Required before Plan 04 migration and Plan 05 seed can run.
    > **Region:** `ap-southeast-3` (Kuala Lumpur) — non-negotiable per MULTI-CLOUD-ARCHITECTURE.md.
    > **Time estimate:** 20-40 minutes (instance provisioning is async, ~5-15 min wait).

    ## Why this is in Phase 1

    The backend has no local Postgres (D-10). Every developer command — `make migrate`,
    `make seed`, `make test`, `make dev` — talks to ApsaraDB. So the instance must
    exist before any other Phase 1 task that touches the DB.

    ## What you'll create

    | Resource | Purpose |
    |----------|---------|
    | ApsaraDB PostgreSQL instance (`pg.n2.medium.2c` or smaller) | The DB engine, in `ap-southeast-3` |
    | Database `gingergig` on that instance | Dev + judging-demo data |
    | Database `gingergig_test` on the same instance | Test runs (Plan 06 fixtures + Plan 04/05 verification) |
    | Database account `gingergig_dev` (read+write on both DBs) | Login credential the backend uses |
    | IP allowlist entry for your dev machine's public IP | So your laptop can reach the DB |

    ## Step-by-step

    ### 1. Pick the right region

    Open https://rdsnext.console.aliyun.com — top-left region picker — choose
    **Asia Pacific SE 3 (Kuala Lumpur)** (`ap-southeast-3`). If you accidentally land
    on Singapore (`ap-southeast-1`), STOP — that's the wrong region for the data tier.

    ### 2. Create instance

    "Create Instance" → choose:
    - Engine: PostgreSQL
    - Version: 14 or higher (14 is fine; 16 if available)
    - Edition: Basic (single node) is sufficient for hackathon demo
    - Storage type: SSD (Cloud SSD or ESSD PL1)
    - Storage: 20 GB (smallest)
    - Instance type: smallest available — `pg.n2.medium.2c` or smaller
    - VPC + zone: any one in ap-southeast-3
    - Network type: classic OR VPC — pick what your dev machine can reach

    Provisioning takes 5-15 minutes. Wait for status `Running`.

    ### 3. Add your IP to the allowlist

    Console → Instance details → **Whitelist Settings** (or "Data Security" → "Whitelist") →
    edit the default group → add your dev machine's egress public IP (find it via
    `curl ifconfig.me`).

    Optional: add a CIDR if you switch networks (but never `0.0.0.0/0` — that's
    public-Internet exposure).

    ### 4. Enable SSL/TLS (D-22)

    Console → Instance details → **SSL Settings** → enable SSL → set to **Required**
    (or "Force"). The asyncpg engine in `app/db/session.py` passes `ssl="require"`
    which only verifies that TLS is used; ApsaraDB doesn't enforce a CA bundle yet.
    Phase 8 may tighten to a verified context with the CA.

    ### 5. Create the database account

    Console → **Accounts** → Create Account.
    - Username: `gingergig_dev`
    - Password: generate a strong random password (≥ 16 chars). Save to your
      password manager.
    - Account type: Privileged Account (or, for tighter control, Standard with
      USAGE+CREATE on both DBs).

    ### 6. Create both databases

    Console → **Databases** → Create Database. Repeat twice:
    - Database name: `gingergig` — Owner: `gingergig_dev` — Encoding: `UTF8`
    - Database name: `gingergig_test` — Owner: `gingergig_dev` — Encoding: `UTF8`

    ### 7. Get the connection string

    Console → Instance details → **Connection Info**. Copy the public endpoint
    hostname (looks like `pgm-xxxxxxxxxxxx.pg.rds.aliyuncs.com`). Port is `5432`.

    Connection string template:
    ```
    postgresql+asyncpg://gingergig_dev:<PASSWORD>@<HOST>:5432/gingergig
    ```

    For the test DB, substitute `gingergig_test` at the end:
    ```
    postgresql+asyncpg://gingergig_dev:<PASSWORD>@<HOST>:5432/gingergig_test
    ```

    ### 8. Fill `backend/.env`

    From `backend/`:
    ```bash
    cp .env.example .env
    ```

    Edit `.env`:
    ```
    DATABASE_URL=postgresql+asyncpg://gingergig_dev:<PASSWORD>@<HOST>:5432/gingergig
    DATABASE_SSL_MODE=require
    TEST_DATABASE_URL=postgresql+asyncpg://gingergig_dev:<PASSWORD>@<HOST>:5432/gingergig_test
    JWT_SECRET=$(python3 -c "import secrets; print(secrets.token_urlsafe(64))")
    CORS_ORIGINS_CSV=http://localhost:5173
    AWS_REGION=ap-southeast-1
    OSS_REGION=ap-southeast-3
    ```

    Confirm `.env` is gitignored:
    ```bash
    grep -F '.env' .gitignore  # should print ".env"
    ```

    ### 9. Smoke test (REQUIRED before continuing)

    From `backend/`, run:

    ```bash
    cd backend && uv run python -c "
    import asyncio, asyncpg, os
    from urllib.parse import urlparse
    async def check(url, label):
        u = url.replace('postgresql+asyncpg://', 'postgresql://')
        conn = await asyncpg.connect(u, ssl='require')
        v = await conn.fetchval('SELECT 1')
        await conn.close()
        assert v == 1, f'{label}: expected 1, got {v}'
        print(f'OK: {label} reachable over TLS')
    async def main():
        # Read .env directly because Settings would force JWT_SECRET etc.
        env = {}
        for line in open('.env'):
            if '=' in line and not line.strip().startswith('#'):
                k, _, v = line.strip().partition('=')
                env[k] = v
        await check(env['DATABASE_URL'], 'gingergig')
        await check(env['TEST_DATABASE_URL'], 'gingergig_test')
    asyncio.run(main())
    "
    ```

    Both databases should report `OK: ... reachable over TLS`. If either fails:

    | Symptom | Fix |
    |---------|-----|
    | `OSError: [Errno 60] Operation timed out` | IP allowlist missing your egress IP |
    | `asyncpg.exceptions.InvalidAuthorizationSpecificationError` | password wrong, or account doesn't have access to that DB |
    | `asyncpg.exceptions.InvalidCatalogNameError: database "gingergig_test" does not exist` | step 6 missed; create both DBs |
    | `ssl required` / TLS error | step 4 missed; enable SSL Required |
    | `getaddrinfo failed` | hostname wrong; re-copy from console |

    ### 10. (Recovery) If dev DB state gets weird

    Per D-12, recovery path:
    ```bash
    cd backend
    uv run alembic downgrade base
    make migrate
    make seed
    ```

    For `gingergig_test` recovery (used by Plan 04/05/06), the same with:
    ```bash
    DATABASE_URL=$TEST_DATABASE_URL uv run alembic downgrade base
    DATABASE_URL=$TEST_DATABASE_URL uv run alembic upgrade head
    ALLOW_SEED=1 DATABASE_URL=$TEST_DATABASE_URL uv run python -m scripts.seed
    ```

    ## Cost note

    A `pg.n2.medium.2c` (2 vCPU, 4 GB RAM) instance with 20 GB SSD in `ap-southeast-3`
    runs roughly USD 50-80/month at on-demand pricing. For the hackathon week, the
    bill should be under USD 20. Tear down the instance after judging if cost is
    a concern.

    ## What this runbook does NOT cover

    - **Tair (Redis):** Phase 6 provisions Tair, not Phase 1 (D-13).
    - **Alibaba OSS bucket:** Phase 4-5 provisions OSS for provider photos and audio.
    - **AWS S3 / Textract / Rekognition / Transcribe:** Phase 4-5 deals with AWS;
      Phase 1 is Alibaba-only.
    - **Production hardening:** read-only replica, automated backups, parameter
      tuning — Phase 8 deploy concerns.
    ```

    No code changes — this is documentation only. Commit this file alongside Plan 07's plan file.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer machine → ApsaraDB control plane | Alibaba Cloud console authentication; outside the application's threat model |
| `backend/.env` (filled with real credentials) → git | Must remain gitignored |
| ApsaraDB allowlist → public Internet | A misconfigured `0.0.0.0/0` allowlist exposes the DB |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | I (Information Disclosure) | `backend/.env` (with real ApsaraDB password) committed to git | mitigate | Plan 01 added `.env` to `.gitignore`. The runbook step 8 instructs the developer to confirm `grep -F '.env' .gitignore`. |
| T-07-02 | T (Tampering) | `0.0.0.0/0` IP allowlist exposing DB to the public internet | mitigate | Runbook step 3 explicitly says "never `0.0.0.0/0`". Phase 8 hardening adds the deployed ECS instance's IP. |
| T-07-03 | I | Database connection over plaintext (no TLS) | mitigate | Runbook step 4 mandates SSL Required at the instance level; `app.db.session.build_engine` passes `ssl="require"` to asyncpg. |
| T-07-04 | E (EoP) | `gingergig_dev` account having superuser-equivalent privileges, allowing damage to other projects on the same instance | accept | Phase 1 single-tenant instance. Phase 8 may split per-environment accounts. |
| T-07-05 | I | Connection string with embedded password ending up in shell history (`history` shows `psql postgres://user:pw@...`) | accept | Documented but not enforced. Developer hygiene. |
| T-07-06 | D (DoS) | ApsaraDB free-tier connection cap (~30) hit by concurrent dev + CI + tests | mitigate | Plan 01's engine builder caps `pool_size=10, max_overflow=5, pool_pre_ping=True`. Runbook recommends `pg.n2.medium.2c` which has higher caps. |
</threat_model>
