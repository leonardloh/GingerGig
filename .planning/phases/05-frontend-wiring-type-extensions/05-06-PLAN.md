<objective>
Wire `ElderVoice` to the Phase 4 voice-to-profile backend without changing the voice UI. Streaming languages (`en-US`, `zh-CN`) use the WebSocket helper with 16 kHz mono signed Int16 PCM frames; batch languages (`ms-MY`, `ta-IN`) use presigned S3 `PUT`, `submitBatchJob`, and polling. Keep the existing `window.SpeechRecognition || window.webkitSpeechRecognition` fallback for browsers without microphone PCM capture or AudioWorklet/Web Audio support.
</objective>

<threat_model>
Frontend risks for this wave:
- **Token handling:** pass `accessToken` from `PrototypeApp` into `ElderVoice`; only `createVoiceStream({ token, language })` may place it in `?token=`, and no token is stored in local/session storage.
- **Wrong route/contract:** implementation is blocked until Phase 4 exposes `/api/v1/voice-to-profile/stream?token=`, `/voice-to-profile/audio-upload-url`, `/voice-to-profile/batch`, and `/voice-to-profile/batch/{jobId}` with tests passing.
- **Audio data leakage:** browser audio goes directly to AWS S3 via presigned URL for batch; no raw audio is sent through ordinary app HTTP endpoints.
- **No UI drift:** preserve voice screen layout, copy, visual step animation, typing fallback, generated listing card, buttons, CSS, and class names.
</threat_model>
