<objective>
Replace the Phase 1 auth stub with a demo-only bearer-token auth shim. The implementation must let the seeded demo users (`siti@gingergig.my`, `amir@gingergig.my`, `faiz@gingergig.my`) obtain usable signed JWTs, let `/auth/me` return the current user's profile, and provide `get_current_user` / `get_current_user_ws` for later protected routes.

This plan intentionally follows `02-CONTEXT.md`: do **not** implement production password verification, bcrypt checks, refresh tokens, persisted sessions, MFA, password reset, rate limiting, or frontend changes.
</objective>

<threat_model>
## Security Threat Model

**Assets:** demo user identity (`sub` claim), `JWT_SECRET`, bearer tokens accepted by protected routes.

**Threats and mitigations:**
- **Algorithm confusion / unsigned token acceptance:** `decode_jwt()` must call `jwt.decode(..., algorithms=["HS256"], options={"require": ["exp", "sub"]})`; tests must reject an `alg=none` token.
- **Missing/forged bearer token:** `get_current_user` and `get_current_user_ws` must return HTTP 401 for missing, malformed, expired, or unknown-user tokens.
- **Accidental production auth scope creep:** auth route tests must prove demo login works without bcrypt/password verification and invalid demo email returns `401 {"status":401,"message":"Invalid credentials"}`.
- **Secret misconfiguration:** keep `Settings.jwt_secret` minimum length validation in `backend/app/core/config.py`; no plan task should weaken this validation.
</threat_model>
