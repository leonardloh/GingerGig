## What this is

GingerGig is a hyperlocal gig platform for Malaysian elders — a mobile-shaped React app where seniors offer services (cooking, crafts, pet sitting, light housekeeping) to nearby requestors, with a "companion" persona for adult children to monitor mum/dad. The frontend is **implemented and runnable on mock data**; this milestone implements a real **multi-cloud FastAPI backend** that the existing typed API client connects to.

**Core constraint (non-negotiable):** Do not change any frontend feature, layout, copy, or styling. The only frontend edits permitted are:
1. Additive type extensions in `frontend/src/services/api/types.ts`
2. Swapping inline mock helpers for `src/services/api/endpoints/*` imports
3. Wiring `ElderVoice`'s WebSocket while keeping the `SpeechRecognition` fallback
4. Updating `VITE_API_BASE_URL` env

## Conventions

**API:** All routes under `/api/v1/*`. Errors return `{status, message, detail?}` matching the frontend's `ApiError`. Bearer token via `Authorization: Bearer <jwt>`.

**Locale-aware data:** `companion_alerts`, `timeline_events`, and `listings.title_*` store all four locales as columns (`*_ms`, `*_en`, `*_zh`, `*_ta`). Project at SQL level based on requesting user's `locale`.

**Booking denormalisation:** `bookings` rows snapshot `requestor_name`, `requestor_initials`, `requestor_avatar_url`, `listing_title`, `quantity_label`, `item_description` so they survive future user/listing edits.

**Demo accounts:** Siti (elder, KYC approved), Amir (requestor), Faiz (companion). All seeded with bcrypt-hashed password `demo`. The prototype's quick-login chips must keep working.

**Seed scripts:** Idempotent via `ON CONFLICT (id) DO UPDATE`. Deterministic IDs (UUID5 namespaced). Re-running the seed twice leaves the DB unchanged.
